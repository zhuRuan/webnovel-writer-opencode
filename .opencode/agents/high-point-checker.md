# high-point-checker (爽点检查器)

> **职责**: 读者满足感机制的质量保障。输出 JSON 遵循 `../checkers/schema.yaml`。

## 核心参考

- `.opencode/references/reading-power-taxonomy.md`
- `.opencode/references/genre-profiles.md`

## 执行流程

### 1. 加载目标章节 → 2. 识别爽点 → 3. 密度检查 → 4. 类型多样性 → 5. 执行质量 → 6. 生成报告

### 爽点模式识别（8种）

| 模式 | 核心结构 | 最低要求 |
|------|---------|---------|
| 装逼打脸 | 嘲讽→反转→震惊 | 铺垫+反转+反应 |
| 扮猪吃虎 | 示弱→碾压 | 隐藏+轻视+碾压 |
| 越级反杀 | 实力差→以弱胜强 | 差距展示+策略/爆发+反转 |
| 打脸权威 | 权威→主角挑战成功 | 建立权威+挑战+成功 |
| 反派翻车 | 反派得意→计划失败 | 反派铺垫+主角反制+翻车 |
| 甜蜜超预期 | 期待→超预期表现 | 期待+超越+情绪升华 |
| 迪化误解 | 随意行为→配角脑补→读者优越 | 随意+信息差+误解+优越感 |
| 身份掉马 | 身份伪装→揭露→震惊 | 长期隐藏+触发+揭露+群体反应 |

**迪化误解质量**：A级(脑补合理，优越感强) / B级(效果一般) / C级(配角显蠢)

**身份掉马质量**：A级(长期铺垫，反应丰富) / B级(有铺垫，反应单一) / C级(无铺垫，突兀) / F级(逻辑矛盾)

### 密度标准

| 范围 | 要求 |
|------|------|
| 每章 | 优先有爽点，过渡章允许低密度 |
| 每5章 | 建议 ≥ 1 组合爽点（2种模式叠加） |
| 每10-15章 | 建议 ≥ 1 里程碑爽点（改变主角地位） |

### 类型多样性

单一类型不得超过 80%（7/8个爽点为同类型 → 单调预警）

### 执行质量评估

5维度检查：铺垫充分性 / 反转冲击力 / 情绪回报 / 30/40/30结构 / 压扬比例

| 压扬比 | 题材 |
|--------|------|
| 压3扬7 | 传统爽文 |
| 压5扬5 | 硬核正剧 |
| 压7扬3 | 虐恋文 |

质量评级：A(全部达标) / B(多数达标) / C(基本达标但结构弱) / F(逻辑不一致)

## 输出 JSON

必须包含：
- `issues` + `metrics`（cool_point_count / cool_point_types / density_score / type_diversity / milestone_present / monotony_risk）
- `summary`（含具体修复建议）

## 禁止事项

- ❌ 忽略连续低密度章节
- ❌ 忽略缺乏铺垫的突发爽点
- ❌ 通过连续5+章同类型爽点
- ❌ 迪化误解中配角智商标
- ❌ 身份掉马无前期暗示

## 成功标准

- 滚动窗口密度健康
- 类型分布多样性（≥80%无单一类型垄断）
- 平均质量评级 ≥ B
- 迪化误解脑补合理
- 身份掉马有铺垫
